En **agent** er fundamentalt anderledes end en assistent:

- **Assistent:** Du spørger, den svarer. Du handler, den hjælper.
- **Agent:** Du definerer målet. Den planlægger, eksekverer og leverer.

## Hvad Gør en Agent Anderledes?

En <Voice text="A-I">AI</Voice>-agent har tre kernekompetencer:

### 1. Planlægning

Givet et mål, bryder agenten det ned i trin:

*"For at tilføje form-validering skal jeg:
1. Identificere signup-komponenten
2. Analysere eksisterende formular-logik
3. Tilføje validerings-funktioner
4. Opdatere <Voice text="U-I">UI</Voice> med fejlbeskeder
5. Skrive tests
6. Oprette Pull Request"*

### 2. Eksekvering

Agenten udfører hvert trin:
- Læser filer
- Skriver kode
- Kører kommandoer
- Verificerer output

### 3. Selvkorrigering

Hvis noget fejler, prøver agenten andre tilgange:

*"Test fejlede. Lad mig analysere fejlmeddelelsen og justere koden."*

## Assign Issues til <Voice text="A-I">AI</Voice>-Agenter

I GitHub kan du nu tildele Issues direkte til <Voice text="A-I">AI</Voice>-agenter som <Voice text="Co-pilot">Copilot</Voice>.

### Sådan gør du

1. **Opret et Issue** med en klar beskrivelse
2. **Tilføj kontekst** — links til relevante filer, eksempler, begrænsninger
3. **Assign til @copilot** (eller anden agent)
4. **Vent** — agenten arbejder i baggrunden

### Hvad sker der bag kulisserne?

Når du tildeler et Issue til <Voice text="Co-pilot">Copilot</Voice>:

```
1. Copilot læser Issue-beskrivelsen
2. Analyserer repository-strukturen
3. Identificerer relevante filer
4. Opretter en branch
5. Implementerer løsningen
6. Kører tests
7. Opretter Pull Request
8. Linker til det originale Issue
```

Alt dette sker uden din involvering.

## Interaktiv Agent Mode: Følg Med i Processen

Selvom agenter kan arbejde autonomt, behøver du ikke vente passivt. I interaktiv mode kan du:

### Observere i realtid

Se hvilke filer agenten redigerer. Følg med i dens tankeproces. Forstå dens beslutninger.

### Godkende eller afvise ændringer

Agenten foreslår en ændring. Du kan:
- **Godkende** — "Ja, implementer det"
- **Afvise** — "Nej, prøv en anden tilgang"
- **Justere** — "Næsten, men ændr X til Y"

### Give feedback undervejs

*"Den tilgang virker for kompleks. Kan du prøve noget simplere?"*

Agenten justerer baseret på din feedback.

## Chat Med Dine Agenter

En af de mest kraftfulde funktioner er muligheden for at *chatte* med agenten, mens den arbejder.

### Præcisér krav

*"Når jeg siger 'password-styrke', mener jeg minimum 8 tegn, mindst ét tal, og mindst ét specialtegn."*

### Stil spørgsmål

*"Hvorfor valgte du at bruge regex fremfor et bibliotek til validering?"*

### Giv kontekst

*"Husk at vi bruger Zod til validering andre steder i projektet. Vær konsistent."*

Dialogen er ikke envejs. Agenten kan også stille *dig* spørgsmål:

*"Jeg fandt to former for email-validering i kodebasen. Hvilken skal jeg følge?"*

## Review og Godkendelse

Når agenten er færdig, oprettes en Pull Request. Nu er det din tur.

### Læs ændringerne

<View>
GitHub viser præcis, hvad der er ændret:
</View>

```diff
+ function validateEmail(email: string): boolean {
+   const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
+   return regex.test(email);
+ }
```

### Verificer kvaliteten

- Er koden læsbar?
- Følger den projektets standarder?
- Er der tests?
- Giver navngivningen mening?

### Kør tests

De automatiske tests (GitHub Actions) fortæller dig, om koden virker. Men du kan også teste lokalt.

### Godkend eller anmod om ændringer

Præcis som med en menneskelig kollega:
- **Approve** — "Ser godt ud, merge det"
- **Request changes** — "Fix dette før merge"

## Tillidsverifikation: Mennesket i Loopet

Her er det vigtige budskab:

> **Agentic <Voice text="A-I">AI</Voice> fjerner ikke behovet for mennesker. Det ændrer vores rolle.**

Du er ikke længere den, der skriver *al* koden. Du er den, der:
- Definerer retningen
- Verificerer kvaliteten
- Træffer de svære beslutninger

Det er som at være arkitekt fremfor murer. Du designer. <Voice text="A-I'en">AI'en</Voice> bygger.

### Hvornår skal du override?

Stol på <Voice text="A-I'en">AI'en</Voice>, men verificer altid:
- **Sikkerhedskritisk kode** — altid manuel review
- **Kernefunktionalitet** — ekstra grundigt review
- **Edge cases** — <Voice text="A-I'en">AI'en</Voice> kan misse dem
- **Forretningslogik** — du kender konteksten bedre
