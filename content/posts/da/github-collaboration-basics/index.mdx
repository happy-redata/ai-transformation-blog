## Repositories: Dit Digitale Værksted

Et **repository** (eller bare "repo") er en mappe til dit projekt — men i skyen. Tænk på det som:

- **En mappe** med al din kode
- **En tidsmaskine** der husker alle ændringer
- **Et samarbejdsrum** hvor flere kan arbejde samtidig

Hos Happy Mates har vi mange repositories:
- `happymates-blob` — denne blog
- `happymates-api-core` — vores <Voice text="A-P-I">API</Voice>
- `happymates-mobile` — vores app

Hvert repository er et selvstændigt projekt med sin egen historie, sine egne samarbejdspartnere, og sin egen automatisering.

## Issues: Opgaver Med Superkræfter

Et **Issue** er mere end en fejlrapport. Det er en arbejdsopgave med indbygget kontekst.

### Hvad kan et Issue indeholde?

| Element | Formål |
|---------|--------|
| **Titel** | Kort beskrivelse af opgaven |
| **Beskrivelse** | Detaljer, krav, kontekst |
| **Labels** | Kategorisering (bug, feature, documentation) |
| **Assignee** | Hvem arbejder på det? |
| **Milestone** | Hvilken version/sprint? |
| **Kommentarer** | Diskussion og opdateringer |

### Issues som "Source of Truth"

Det smarte ved Issues er, at de samler alt på ét sted:

- Hvad skal laves?
- Hvem laver det?
- Hvad er status?
- Hvad blev diskuteret?

Når du lukker et Issue, har du en komplet historie af opgaven — fra idé til løsning.

## Pull Requests: Forslag Til Ændringer

En **Pull Request** (eller <Voice text="P-R">PR</Voice>) er et forslag om at ændre koden. Det er som at sige:

*"Hej team, jeg har lavet noget. Vil I kigge på det og godkende, at det bliver en del af projektet?"*

### Anatomien af en Pull Request

1. **Titel og beskrivelse** — Hvad og hvorfor
2. **Ændringer** — Præcis hvilke filer der er redigeret
3. **Review** — Kollegaer gennemgår koden
4. **Diskussion** — Kommentarer på specifikke linjer
5. **Godkendelse** — Grønt lys fra reviewere
6. **Merge** — Ændringerne bliver en del af projektet

### Hvorfor ikke bare ændre direkte?

Forestil dig et team på 10 udviklere, der alle ændrer i den samme fil samtidig. Kaos!

Pull Requests giver:
- **Kvalitetskontrol** — Nogen kigger på koden før den går i produktion
- **Dokumentation** — En historie af hvad der blev ændret og hvorfor
- **Rollback-mulighed** — Hvis noget går galt, kan vi nemt gå tilbage

## GitHub Actions: Den Automatiske Fabrik

**GitHub Actions** er automatisering. Når noget sker i dit repository, kan du udløse handlinger.

### Hvad kan Actions gøre?

| Trigger | Handling |
|---------|----------|
| Nyt commit pushes | Kør tests automatisk |
| Pull Request oprettes | Check kode-kvalitet |
| <Voice text="P-R">PR</Voice> merges til main | Deploy til produktion |
| Dagligt kl. 9 | Kør database-backup |

### Et simpelt eksempel

Hos Happy Mates kører vi automatisk:
- **Linting** — Checker at koden følger vores standarder
- **Tests** — Verificerer at alt virker
- **Build** — Bygger produktionsversionen
- **Deploy** — Sender til serveren

Alt sammen uden at vi løfter en finger.

```yaml
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build
      - name: Deploy
        run: npm run deploy
```

Hver gang vi pusher til `main`, bygges og deployes bloggen automatisk.

## Hvordan Det Hele Hænger Sammen

Her er den typiske workflow i et professionelt projekt:

### 1. Opret et Issue
*"Tilføj dark mode til forsiden"*

### 2. Opret en Branch
En "gren" af koden, hvor du kan eksperimentere uden at ødelægge noget.

### 3. Lav dine ændringer
Skriv kode, test lokalt.

### 4. Opret en Pull Request
*"Her er min implementation af dark mode"*

### 5. Code Review
Kollegaer kigger på koden, stiller spørgsmål, foreslår forbedringer.

### 6. Juster efter feedback
Måske fanger revieweren en fejl. Du retter den.

### 7. Godkend og Merge
Revieweren godkender. Koden bliver en del af projektet.

### 8. Actions køres
Automatisk test, build, deploy.

### 9. Issue lukkes
Opgaven er færdig. Historikken er dokumenteret.

<View>
Flowet ser sådan ud:
</View>

```
Issue → Branch → PR → Review → Merge → Action → Lukket Issue
```

## Kom i Gang med GitHub

### 1. Opret en konto
Gå til [github.com](https://github.com) og tilmeld dig (gratis).

### 2. Opret dit første repository
Klik "New repository", giv det et navn, og du er i gang.

### 3. Prøv GitHub Desktop
Hvis terminalen virker skræmmende, er [GitHub Desktop](https://desktop.github.com/) en visuel måde at arbejde med Git.

Opret et Issue i dit eget repository. Prøv at lukke det med en Pull Request.
