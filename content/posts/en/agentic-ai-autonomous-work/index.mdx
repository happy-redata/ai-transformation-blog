An **agent** is fundamentally different from an assistant:

- **Assistant:** You ask, it answers. You act, it helps.
- **Agent:** You define the goal. It plans, executes, and delivers.

## What Makes an Agent Different?

An <Voice text="A-I">AI</Voice> agent has three core competencies:

### 1. Planning

Given a goal, the agent breaks it down into steps:

*"To add form validation, I need to:
1. Identify the signup component
2. Analyze existing form logic
3. Add validation functions
4. Update <Voice text="U-I">UI</Voice> with error messages
5. Write tests
6. Create Pull Request"*

### 2. Execution

The agent performs each step:
- Reads files
- Writes code
- Runs commands
- Verifies output

### 3. Self-Correction

If something fails, the agent tries other approaches:

*"Test failed. Let me analyze the error message and adjust the code."*

## Assigning Issues to <Voice text="A-I">AI</Voice> Agents

In GitHub, you can now assign Issues directly to <Voice text="A-I">AI</Voice> agents like <Voice text="Co-pilot">Copilot</Voice>.

### How to do it

1. **Create an Issue** with a clear description
2. **Add context** — links to relevant files, examples, constraints
3. **Assign to @copilot** (or another agent)
4. **Wait** — the agent works in the background

### What happens behind the scenes?

When you assign an Issue to <Voice text="Co-pilot">Copilot</Voice>:

```
1. Copilot reads the Issue description
2. Analyzes the repository structure
3. Identifies relevant files
4. Creates a branch
5. Implements the solution
6. Runs tests
7. Creates Pull Request
8. Links to the original Issue
```

All of this happens without your involvement.

## Interactive Agent Mode: Follow Along

Even though agents can work autonomously, you don't have to wait passively. In interactive mode, you can:

### Observe in real-time

See which files the agent is editing. Follow its thought process. Understand its decisions.

### Approve or reject changes

The agent proposes a change. You can:
- **Approve** — "Yes, implement it"
- **Reject** — "No, try a different approach"
- **Adjust** — "Almost, but change X to Y"

### Provide feedback along the way

*"That approach seems too complex. Can you try something simpler?"*

The agent adjusts based on your feedback.

---

## Chat With Your Agents

One of the most powerful features is the ability to *chat* with the agent while it works.

### Clarify requirements

*"When I say 'password strength', I mean minimum 8 characters, at least one number, and at least one special character."*

### Ask questions

*"Why did you choose to use regex instead of a library for validation?"*

### Provide context

*"Remember that we use Zod for validation elsewhere in the project. Be consistent."*

The dialogue isn't one-way. The agent can also ask *you* questions:

*"I found two forms of email validation in the codebase. Which should I follow?"*

## Review and Approval

When the agent is done, a Pull Request is created. Now it's your turn.

### Read the changes

<View>
GitHub shows exactly what was changed:
</View>

```diff
+ function validateEmail(email: string): boolean {
+   const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
+   return regex.test(email);
+ }
```

### Verify quality

- Is the code readable?
- Does it follow project standards?
- Are there tests?
- Does the naming make sense?

### Run tests

The automatic tests (GitHub Actions) tell you if the code works. But you can also test locally.

### Approve or request changes

Exactly like with a human colleague:
- **Approve** — "Looks good, merge it"
- **Request changes** — "Fix this before merge"

## Trust Verification: The Human in the Loop

Here's the important message:

> **Agentic <Voice text="A-I">AI</Voice> doesn't remove the need for humans. It changes our role.**

You're no longer the one writing *all* the code. You're the one who:
- Defines the direction
- Verifies quality
- Makes the tough decisions

It's like being an architect instead of a bricklayer. You design. The <Voice text="A-I">AI</Voice> builds.

### When should you override?

Trust the <Voice text="A-I">AI</Voice>, but always verify:
- **Security-critical code** — always manual review
- **Core functionality** — extra thorough review
- **Edge cases** — the <Voice text="A-I">AI</Voice> might miss them
- **Business logic** — you know the context better
